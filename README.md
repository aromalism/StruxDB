# StruxDB

A lightweight, file-based database manager written in C. StruxDB lets users register accounts, log in, and manage personal datasets — creating custom schemas, adding entries, and searching records — all from an interactive command-line interface.

---

## Features

- User registration with enforced password policy (length, complexity, no whitespace)
- Secure login with per-user isolated workspaces
- Create named datasets with custom field schemas
- Add, view, and search entries within any dataset
- All data stored locally under `data/`

---

## Requirements

- GCC (or any C99-compatible compiler)
- GNU Make
- POSIX-compatible OS (Linux, macOS)

---

## Build

Clone the repository and run Make from the project root:

```bash
git clone <repo-url>
cd struxdb
make
```

The compiled binary is placed at `bin/dbms`.

To clean build artifacts:

```bash
make clean
```

---

## Run

```bash
./bin/dbms
```

The application will create the `data/` directory automatically on first run.

## Usage

### Registration

1. Select **Register** from the main menu.
2. Choose a username (3–32 characters; letters, numbers, underscores, and periods only; cannot start with a digit or underscore).
3. Set a password that meets all of the following:
   - 8–128 characters
   - At least one uppercase letter
   - At least one lowercase letter
   - At least one digit
   - At least one symbol
   - No whitespace
   - Not the same as your username

### Login

Select **Login** and enter your registered credentials. On success you are taken to your personal workspace.

### Workspace

Each logged-in user has an isolated workspace. From there you can:

| Option | Description |
|--------|-------------|
| List datasets | Show all dataset files in your workspace |
| Create dataset | Define a new dataset with a custom field schema |
| Open dataset | Open an existing dataset to add or search entries |
| Exit | Return to the main menu |

### Datasets

Datasets use a pipe-delimited plain-text format. When creating a dataset you define the field names (e.g. `name,phone,email`). These become the schema for every entry in that file.

Inside an open dataset:

| Option | Description |
|--------|-------------|
| Add entry | Prompt for each field and append a new record |
| Search entries | Full-text search across all records |
| Show all entries | Display every record with field labels |
| Back | Return to the workspace menu |

## Data Storage

| Path | Contents |
|------|----------|
| `data/credentials.txt` | Usernames and passwords, one per line |
| `data/users/<username>/` | Directory holding that user's dataset files |
| `data/users/<username>/*.strux` | Pipe-delimited dataset files (plaintext, current) |

> **Warning:** Passwords and dataset entries are currently stored in plaintext. Do not use real or sensitive credentials or data. See [Known Limitations](#known-limitations) below.

---

## Known Limitations

- **Plaintext password storage** — credentials are written to disk without hashing. A future release will use a library such as `libsodium` (`crypto_pwhash`) for secure password storage.
- **Plaintext dataset storage** — `.strux` files are currently readable pipe-delimited text. The planned `.strux` binary format (see [Future Goals](#future-goals)) will replace this with encrypted byte data.
- **No delete or update** — entries can be added and searched, but not edited or removed after creation.
- **Single-user sessions** — there is no concurrent access control; simultaneous writes to the same dataset from multiple processes are unsafe.
- **POSIX only** — the `mkdir` calls and directory API (`<dirent.h>`) require a POSIX environment; Windows is not supported.

---

## Future Goals

### Password Hashing

Plain-text credential storage is a known limitation. The planned fix is to hash passwords before writing them to disk using **libsodium's `crypto_pwhash`** (Argon2id), which provides memory-hard hashing resistant to brute-force and GPU attacks. Login will use `crypto_pwhash_str_verify` instead of `strcmp`.

### `.strux` — Encrypted Binary File Format

The current pipe-delimited `.strux` files will be replaced by a proper binary format of the same name. The goals for this format are:

- **Encryption at rest** — all field data encrypted with **AES-256-GCM** (authenticated encryption) before being written to disk. The encryption key will be derived from the user's password via a KDF (e.g. Argon2id), meaning no separate key file is needed — only the correct password can decrypt the data.
- **Binary encoding** — records stored as structured byte sequences rather than human-readable text, making the file opaque to casual inspection and resistant to manual tampering.
- **File header** — a fixed-size binary header at the start of each `.strux` file will carry: a magic number to identify the format, a version byte, the KDF salt, the AES nonce/IV, and the field count and field-name block.
- **Integrity verification** — GCM's authentication tag will detect any corruption or tampering of the ciphertext before decryption is attempted.

Rough layout of a `.strux` file (planned):

```
[ magic: 6 bytes  ] "STRUX\0"
[ version: 1 byte ]
[ kdf_salt: 16 bytes ]
[ aes_nonce: 12 bytes ]
[ field_count: 2 bytes ]
[ field_names: variable, null-delimited ]
--- encrypted payload (AES-256-GCM) ---
[ record_count: 4 bytes ]
[ records: variable-length binary rows ]
[ auth_tag: 16 bytes ]
```

### Networking

A future networking layer will allow StruxDB to operate in a client–server model:

- A **server process** will host one or more user databases and listen on a configurable TCP port.
- A **client CLI** will connect to the server, authenticate over an encrypted channel (TLS via **mbedTLS** or **OpenSSL**), and issue commands (add, search, list) remotely.
- **Per-session tokens** will replace password transmission after the initial handshake, reducing credential exposure on the wire.
- A simple **text-based protocol** (or optionally a compact binary protocol) will define the command/response framing between client and server.
